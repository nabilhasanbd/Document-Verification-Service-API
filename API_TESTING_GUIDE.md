# Document Verification Service — API Testing Guide

Base URL: `http://localhost:8080`

> How to run the project:
> ```powershell
> docker compose up -d          # start Postgres + Redis + LocalStack S3
> docker exec docverify-redis redis-cli FLUSHDB   # (optional) clear old cache
> ./mvnw.cmd spring-boot:run    # start the app
> ```
>
> ⚠️ Make sure `application.yml` datasource matches `docker-compose.yml`
> (`docverify` / `docverify` / `docverify`).

## 1. Create Customer

`POST /api/customers`

**Request JSON**
```json
{
  "name": "Rahim Uddin",
  "email": "rahim@example.com",
  "phone": "+8801712345678"
}
```

**Example (curl)**
```bash
curl -X POST http://localhost:8080/api/customers \
  -H "Content-Type: application/json" \
  -d '{"name":"Rahim Uddin","email":"rahim@example.com","phone":"+8801712345678"}'
```

**Response (201 Created)**
```json
{
  "id": "2861d001-2c6b-486d-9a9c-0ff755c996c1",
  "fullName": "Rahim Uddin",
  "email": "rahim@example.com",
  "phone": "+8801712345678",
  "createdAt": "2026-07-16T01:00:00Z",
  "updatedAt": "2026-07-16T01:00:00Z",
  "documents": []
}
```

> Save the returned `id` — you will use it as `{customerId}` everywhere below.

## 4. Generate Pre-signed Upload URL

`POST /api/customers/{customerId}/documents/presigned-url`

**Request JSON**
```json
{
  "fileName": "national-id.jpg"
}
```

**Example**
```bash
curl -X POST http://localhost:8080/api/customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/documents/presigned-url \
  -H "Content-Type: application/json" \
  -d '{"fileName":"national-id.jpg"}'
```

**Response (200 OK)**
```json
{
  "uploadUrl": "http://localhost:4566/doc-verification-bucket/customers/2861d001-.../documents/a3f2b1c0-...-national-id.jpg?X-Amz-...",
  "fileKey": "customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/documents/a3f2b1c0-...-national-id.jpg",
  "expiresIn": 600
}
```

> Save the returned `fileKey` — you will use it in the confirm step.
> `uploadUrl` is a LocalStack S3 pre-signed PUT URL (valid for 10 minutes).
>
> The `fileKey` follows the format `customers/{customerId}/documents/{uuid}-{fileName}`
> (generated by `S3ServiceImpl.generateObjectKey`). Copy it **exactly** from the response.

## 5. Upload File to S3

This is **not** one of our APIs — it uploads the actual file directly to S3
(LocalStack) using the `uploadUrl` from step 4.

**Example (upload a local file)** — use the full `uploadUrl` returned by step 4:
```bash
curl -X PUT \
  --data-binary @"./national-id.jpg" \
  "http://localhost:4566/doc-verification-bucket/customers/2861d001-.../documents/a3f2b1c0-...-national-id.jpg?X-Amz-Algorithm=..."
```

> Or, create a dummy file first:
> ```bash
> echo "dummy document content" > national-id.jpg
> ```
> A successful upload returns no body (HTTP 200).

## 6. Confirm Upload

`POST /api/customers/{customerId}/documents/confirm-upload`

After the file is in S3, confirm the upload. This:
- creates a `Document` (status = `UPLOADED`)
- creates a `Verification` (status = `PENDING`)
- caches the status `PENDING` in Redis

**Request JSON**
```json
{
  "fileKey": "customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/documents/a3f2b1c0-...-national-id.jpg"
}
```

**Example**
```bash
curl -X POST http://localhost:8080/api/customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/documents/confirm-upload \
  -H "Content-Type: application/json" \
  -d '{"fileKey":"customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/documents/a3f2b1c0-...-national-id.jpg"}'
```

**Response (200 OK)**
```json
{
  "documentId": "4ed05ebb-b962-4bfb-8f98-6c6da28026f6",
  "fileKey": "customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/documents/a3f2b1c0-...-national-id.jpg",
  "fileName": "national-id.jpg",
  "contentType": "application/octet-stream",
  "size": 23,
  "status": "UPLOADED",
  "uploadedAt": "2026-07-16T01:05:00Z"
}
```

> If the object is not found in S3, you get `400 Bad Request`:
> ```json
> { "message": "Uploaded object not found in S3 for key: ..." }
> ```

## 7. Get Verification Status

`GET /api/customers/{customerId}/verification-status`

Returns the **verification** status. Cache-Aside flow:
- First checks Redis.
- If present → returns it immediately (cache hit).
- If missing → loads latest from DB, caches it, returns it.

**Example**
```bash
curl http://localhost:8080/api/customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/verification-status
```

**Response right after confirm-upload (200 OK)**
```json
{
  "documentId": "4ed05ebb-b962-4bfb-8f98-6c6da28026f6",
  "status": "PENDING"
}
```

> Verify the Redis cache:
> ```bash
> docker exec docverify-redis redis-cli GET "verification-status:2861d001-2c6b-486d-9a9c-0ff755c996c1"
> ```
> Expected value: `{"documentId":"4ed05ebb-...","status":"PENDING"}`

## 8. Run External Verification

`POST /api/customers/{customerId}/verify`

Calls the (mock) external provider. On result it:
- updates the `Verification` table
- updates the `Document` status (if completed)
- **evicts** the Redis cache entry
- **saves** the new status back into Redis immediately

**Example (no request body)**
```bash
curl -X POST http://localhost:8080/api/customers/2861d001-2c6b-486d-9a9c-0ff755c996c1/verify
```

**Response (200 OK) — when approved**
```json
{
  "id": "a1b2c3d4-...",
  "customerId": "2861d001-2c6b-486d-9a9c-0ff755c996c1",
  "documentId": "4ed05ebb-b962-4bfb-8f98-6c6da28026f6",
  "status": "VERIFIED",
  "providerReference": "MOCK-9f8e7d6c-...",
  "remarks": "Verification successful",
  "verifiedAt": "2026-07-16T01:10:00Z",
  "createdAt": "2026-07-16T01:05:00Z"
}
```

Possible `status` values:

| Provider result | Verification status |
|-----------------|---------------------|
| APPROVED        | `VERIFIED`          |
| REJECTED        | `REJECTED`          |
| PENDING (fallback) | `FAILED` *(see note)* |
| anything else   | `FAILED`            |

> The mock provider approves ~85% of the time and is unavailable ~20% of the time.
> When unavailable, the client fallback returns provider status `"PENDING"`, but
> because `mapVerificationStatus` has no `"PENDING"` case, it falls through to the
> `default` branch → the **verification becomes `FAILED`** (not PENDING).
> The document status stays `VERIFICATION_IN_PROGRESS` (since the result is not a
> terminal APPROVED/REJECTED). Run step 8 again to retry.

## Status Reference

**Verification status** (from `verifications` table — what the status API returns):

| Status                   | Meaning                                        |
|--------------------------|------------------------------------------------|
| `PENDING`                | Verification created, not yet processed        |
| `VERIFIED`               | Provider approved                              |
| `REJECTED`               | Provider rejected                              |
| `FAILED`                 | Provider returned an unknown/unexpected result |

**Document status** (from `documents` table):

| Status                       | Meaning                                     |
|------------------------------|---------------------------------------------|
| `UPLOADED`                   | File confirmed in S3                        |
| `VERIFICATION_IN_PROGRESS`   | Verification running / provider unavailable |
| `VERIFIED`                   | Verification approved                       |
| `REJECTED`                   | Verification rejected                       |

## Cache Key Reference

Redis key format: `verification-status:{customerId}`

Example:
```
verification-status:2861d001-2c6b-486d-9a9c-0ff755c996c1
```

Value (JSON):
```json
{"documentId":"4ed05ebb-b962-4bfb-8f98-6c6da28026f6","status":"PENDING"}
```

Useful Redis commands:
```bash
# read a cached status
docker exec docverify-redis redis-cli GET "verification-status:2861d001-2c6b-486d-9a9c-0ff755c996c1"

# list all verification cache keys
docker exec docverify-redis redis-cli --scan --pattern "verification-status:*"

# clear everything (use when changing serializer format)
docker exec docverify-redis redis-cli FLUSHDB
```
